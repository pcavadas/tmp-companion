# TMP Protocol Quick Reference

Snapshot of currently-known values across the TMP ecosystem. Verify these are still accurate after firmware/app updates — the re-verify recipe at the end of this file tells you how to re-derive each one if it changes.

**Version baseline:** Pro Control v1.7.2, firmware v1.7.75; firmware v1.8.45 deltas are called out inline where verified.

## Preset format

- **XOR key**: `0x4A 0x4C 0x44` (ASCII `JLD`) — same operation encodes and decodes
- **Verify**: decoded result must be valid JSON starting with `{` and ending with `}`
- **If key changes**: re-derive the 3-byte XOR key by known-plaintext (the `{"` prefix) or Kasiski analysis (key length 3) + frequency analysis

## USB

- **VID**: `0x1ed8` (Fender)
- **PID**: `0x0044`
- **HID report size**: 64 bytes, vendor usage `0xFF00`

## Protocol Buffers

- **Top-level wrapper**: `FenderMessageTMS` with 13 category sub-messages
- **Key messages**: `ChangeParameter`, `LoadScene` (PresetMessage field 101, addresses the CURRENT preset only — encoder golden-tested in the companion's `proto.rs`), `SetNodeSceneEdit`, `SetPresetLevel`, `SaveCurrentPreset`, `CurrentPresetDataJsonRequest` (burst-only — see retrieval gotcha below)
- **Scene addressing (HW-proven 2026-06-04, REFUTES the earlier "1-based" note)**: `LoadScene.sceneSlot` / `SceneLoaded.sceneSlot` / `lastLoadedScene` are **0-based `scenes[]` indices**; the BASE scene is the **CONSTANT wire slot 8** — even for a 0-scene preset (a 0-scene preset's base `SceneLoaded` echoed slot 8). NOT `scene_count + 1`. The encoder **MUST emit `sceneSlot` explicitly even for 0** — the device IGNORES an empty `LoadScene{}` (proto3 zero-omission left `scenes[0]` unloadable until fixed). Proof: `probe --loadscene 1` activated `scenes[1]` ("Reverb"). ⚠️ OPEN anomaly: on a 2-amp preset, USB `loadScene(0)` materialized a DIFFERENT amp state than the physical footswitch tap (companion per-scene leveling is RED on scene 0).
- **Firmware version retrieval**: send `currentFwRequest` (SettingsMessage field 1, `dummy=true`) **in-burst, with NO batchStatus, in the batch-2 group — after `userir` but BEFORE `CurrentPresetDataRequest`** → `currentFwResponse` (SettingsMessage field 2) `data` (field 1) = e.g. `"1.7.75"`. Sending it AFTER the batch-3 `CurrentPresetDataRequest` makes the device drop the reply; standalone after the burst → `ConnectionError`. (Live-confirmed 2026-06-03; an earlier "arrives unsolicited" observation was a one-off that did not reproduce after a reboot.)
- **Preset JSON retrieval**: the preset JSON arrives as `CurrentPresetDataChanged` (PresetMessage field 3, LZ4-block), elicited by `CurrentPresetDataJsonRequest` (field 78) sent **inside the handshake burst right after `CurrentPresetDataRequest` (field 2)** — a standalone request later gets NO reply. No complete-preset-JSON path over USB: field 79 (`CurrentPresetDataJsonResponse`) appears unimplemented; field 136 = `AllBlockPresetsResponse` (block-preset library, not preset data). **The ~3.4 KB truncation is SESSION-HEALTH-DEPENDENT, not a hard limit (verified 2026-06-04):** a lean/stale connection truncates at `"scenes"`, but a HEALTHY dense-heartbeat (~250 ms) session delivers ~17 KB — the complete `scenes[].sceneName` (slot-ordered) + root `lastLoadedScene` (0-based; base=8) + the full `ftsw` map — truncating only inside the FINAL scene's `uuid`. So **scene NAMES and the live scene index ARE readable live over USB** (no offline library): from the field-3 JSON on a dense session, and `sceneListResponse` (125) pushes the names on every preset SWITCH. (`audioGraph.guitarNodes` survives even the lean prefix via tolerant brace-match.) The companion reads scenes live from the monitor's field-3 decode; the old offline `preset_scenes_all` command was DELETED. **fw 1.8.45 caveat (HW-verified 2026-06-05): field-78 actively HURTS first-paint reads** — its multi-packet reply drowns the field-3 push the lean handshake's own `CurrentPresetDataRequest` already elicits (the JSON prefix carries the full audioGraph within ~1 s). For connect-time graph reads, stay LEAN (no field-78) and decode field-3 from the 0x35-is-final reassembler (`push_bodies`), not the non-final `streams()` rule. **Stronger 2026-06-06 finding: a field-78 session delivers field-3 NEVER on 1.8.45** (3/3 fresh handshakes + a re-load + dense-heartbeat coax all came up empty) — field-3 is pushed only on a CHANGE or at connect, so block/graph discovery must use a rich lean session reading push bodies (companion `prepass_scene_docs` pattern), not field-78. **The field-3 graph push carries NO slot/list-index** — the active list-index arrives via `PresetLoaded` (11), which LAGS field-3 in a preset-switch burst, so a consumer treats the `PresetLoaded`-derived index (companion frontend `activeListIndex`) as the authoritative active slot, NOT the graph (backend slot-stamping at field-3 time is racy). Non-destructive live field-3 capture for HW verification: `probe --dump-currentpresetdata <slot>` (loads the slot, dumps the decompressed JSON, reports scenes/ftsw completeness).
- **Slot-addressed preset read + the connection_request RE-ARM (HW-proven 2026-06-06, fw 1.8.45)**: `presetDataRequest` (PresetMessage field **8**, `{listEnum, slot}` 1-based, NO batchStatus) → `presetDataChanged` (field **9**, `presetJson` = field 3, **plaintext** not LZ4) is a per-slot-deterministic PARTIAL (cut inside the final scene; scene names + `ftsw` survive). The device answers exactly **ONE data request per burst state** — re-sending `connection_request` **RE-ARMS** that state on the open connection, enabling whole-library sweeps on ONE session (26/26 reliable, ~0.5–0.9 s/slot). Rules: fire reads on a **quiet line** (a read sent mid-flood is dropped DEVICE-side — drain the handshake first); the re-arm is REQUIRED on a quiet/dedicated session but HARMFUL on an ALREADY-LIVE session (the monitor's dense ~250 ms heartbeat already armed it) — re-arming a live session makes the device emit a `connectionError` on the next heartbeat (HW 2026-06-16: 1/read + ~140 ms; skip the re-arm there → 0/read, 10/10, same payload; the companion's `read_slot_preset_json_live` does this, oracle `probe --slotread-live`); the field-9 stream ends in a `0x35` final frame so the harvest must use the 0x35-is-final reassembly rule; a batch-bearing `preset_list_request` is NOT answered on a minimal burst (only inside the recognized full handshake).
- **Scene mode**: `SetNodeSceneEdit { nodeId, groupId, sceneEditEnable: true }` enables per-scene parameter control. `nodeId` here is the live `ChangeParameter` coordinate and may differ from the stable Fender model id; classify amps by `FenderId` / companion `model_id`, then write the active block's `nodeId`. **Sequencing (Pedro, HW 2026-06-09):** scene mode gives FULL isolation (a scene's write does NOT touch the base), and ENABLING it is harmless on its own. The leak is purely about ORDER: a value WRITE that lands BEFORE scene mode is enabled+confirmed hits the BASE scene, which every un-scene-moded slot then inherits. `SetNodeSceneEdit` is fire-and-forget (no ack), so give it a solid settle to land before the write (a 300 ms window raced on a busy HID line; use ~600 ms). The BASE scene's blocks have NO scene mode. As a CLEANLINESS practice (not safety), only CHANGE a scene's value when it needs it — measure as-is first and skip scenes already on target.
- **Proto count**: 379 files (Pro Control v1.7.2, SDK v3.13.0)

## USB HID wire envelope (live-confirmed 2026-05-11)

```
host → device   (IOHIDDeviceSetReport, 63-byte payload, no report-id prefix):
    0x35 0x00 <body_len> <protobuf FenderMessageTMS>       padded to 63 with 0x00

device → host   (IOHIDDevice input-report callback, 64 bytes incl. report-id 0):
    0x00 0x35 0x00 <len> <body>            single-frame small response
    0x00 0x33 0x00 0x3c <60 bytes>         stream start (multi-packet response)
    0x00 0x34 0x00 0x3c <60 bytes>         stream continuation chunk
```

- **Host magic**: 0x35 for small bodies — but the host **also chunks outbound**, symmetric with the device: a body > 60 B is sent as `0x33` (start) / `0x34` (continuation) / `0x35` (final) frames, each `MAGIC 0x00 <len> <≤60 B>`. Pro Control uses this for large requests like `importPresetRequest`. **Device magic**: 0x35 / 0x33 / 0x34 (single, stream-start, stream-cont).
- **Reassembler**: 0x33 begins a logical stream; concatenate 60-byte bodies of subsequent 0x34 chunks. **`0x35` is the FINAL frame of that stream — it carries payload (often small) and terminates it; a `0x34` never follows a `0x35` within a stream.** ⚠️ A reassembler that treats an inbound `0x35` as a _standalone_ single-frame (instead of appending it as the final frame) **drops the last frame → systematic tail truncation** of multi-packet list responses (last record's name cut, or whole trailing records dropped). HW-confirmed 2026-06-02 on the song/setlist list reads.
- **Reading multi-packet responses reliably**: parse **fail-closed** — reject any reassembled body with a truncated length-delimited protobuf field (declared length > bytes present), and only accept a response that parses _complete_ to the end; then **retry** the read. A tolerant parser silently clips a truncated field into a plausible-but-wrong record (e.g. a cut song name), so "largest prefix" is unsafe. Multi-packet responses also truncate when the device **interleaves a concurrent `0x33` broadcast stream** (no per-stream ids to demux) — worst right after a write; retry until a complete one lands.

### First-connect handshake (Pro Control 1.7.2)

ConnectionRequest → PresetListRequest(listEnum=1) → FavoriteListRequest → PresetListRequest(listEnum=4) → PresetListRequest(listEnum=3) → ProductProfileRequest → CurrentPresetInfoRequest → SettingsMessage(field 66) → UserIRMessage(field 2) → CurrentPresetDataRequest, then `ConnectionHeartbeat` every ~1.5 s.

**Connect cost (companion measurements, fw 1.7.75/1.8.45):** the full handshake is ~660 ms of fixed pumping (200 ms settle + 8×20 ms inter-send + 300 ms read windows), and a per-command release→fresh-connect→work→400 ms-settle→reconnect bookend totals ~2 s — for repeated commands, multiplex onto a live dense-heartbeat session (see the unsolicited-push gotcha below) instead of reconnecting per command.

### Wire-format gotchas

- **The HID is single-seize / exclusive-open — serialize ALL device access.** Only one client may hold the device. A second _concurrent_ `IOHIDDeviceOpen` (or acquire-before-release on reconnect) throws `kIOReturnExclusiveAccess` (`0xe00002c5`); `kIOReturnNotPrivileged` is `0xe00002e2`. Any driver must funnel every read/write through one seized handle, **release-before-acquire** on reconnect, and **never fire two device commands concurrently** (e.g. a `Promise.all` of two reads, or two parallel connections each via its own seize, collide — a real bug 2026-06-02). This is the OS-level seize invariant, distinct from the message-level `batchStatus` single-in-flight rule. **Post-close lockout (HW 2026-06-06):** a re-open ≤~800 ms after a close succeeds; beyond that the device locks out exclusive opens for tens of seconds and **failed attempts RESET the lockout** — never hammer-retry; re-enumerate at long quiet backoff (details in the companion `CLAUDE.md` HID open-lockout bullet).
- **`ConnectionHeartbeat` must NOT carry `batchStatus`** — adding `pb_field(10, 0, n)` makes the device reply with `ConnectionError`. Bytes are exactly `35 00 06 22 04 22 02 08 01`.
- **Empty `ConnectionError` (`22 02 1a 00`) is the heartbeat ACK**, NOT an error. `errorEnum=0` (the proto3 default) is "no error"; the schema's name is misleading.

#### Control-plane writes (live-confirmed via the auto-leveller, firmware 1.7.75)

- **All fire-and-forget setters/commands MUST omit `batchStatus`** — not just the heartbeat. `SetReAmpMode`, `SetPresetLevel`, `LoadPreset`, `SaveCurrentPreset`, `ChangeParameter`: with field 10 present the device **silently ignores** the command (no error, no effect). Only request/response messages carry `batchStatus`. (This was the single most recurring bug when writing the control plane.)
- **Re-amp toggle** = `SettingsMessage(3) → reampModeActive(30)`. ON = `1a 05 f2 01 02 08 01`, OFF = `1a 03 f2 01 00`. The `ReAmpModeChanged` echo is unreliable — a finite captured USB-Out loudness is the real proof it engaged. ⚠️ **Stranded re-amp = the TMP produces NO SOUND** from the guitar input (input is routed to USB-In while engaged): a fire-and-forget OFF can get dropped under connection churn, leaving the unit silent. Recover with an explicit OFF over USB (companion `probe --reamp-off`) — always send a guaranteed OFF on a fresh connection at the end of any re-amp run.
- **Re-amp latches the preset's state at engage** (set `presetLevel` before engaging for preset leveling; set only the active amp's exact-case `outputLevel` before engaging for scene leveling) and **engages reliably only once per connection** — fresh-connect per engage; repeated in-session toggling is flaky. ⚠️ **DANGER (HW 2026-06): re-engaging re-amp on a HELD connection (disengage→settle→re-engage) once WEDGED the device's re-amp AND triggered a USB crash that REBOOTED the Mac.** The held-session multiplexing win that makes structural edits (replaceNode/removeNode) fast does NOT transfer to re-amp — re-amp MUST reconnect per measurement; never re-engage on a held connection. **Scene granularity (HW 2026-06-06, fw 1.8.45):** the latch covers the ACTIVE SCENE — `loadScene` mid-engage is INAUDIBLE (the captured audio never changes), but `changeParameter` mid-engage IS audible (live knob nudges work) → per-scene leveling needs one engage per scene, Scene Edit on the active amp, and amp `outputLevel` changes only. Never use preamp/master/volume params (`brightvolume`, `mastervolume`, `normalvolume`, `volume`) for scene leveling because they alter tone; fw 1.7.75/1.8.45 embedded amp schemas show every amp has `outputLevel`, `output` is not an amp control, and `level` is not the output leveling control (`ACD_TMRumbleV3` has `level` as a separate amp knob plus `outputLevel`, and `level` must not be changed). Preset 001 Klon proved the first active volume-like param can falsely clamp while Hiwatt `outputLevel=1.0` measures `-9.610 LUFS`. `presetLevel` is only the global fallback: if active amp `outputLevel` cannot hit target, raise `presetLevel` and restart scene leveling from scene 0; see the companion `CLAUDE.md` BatchedLive bullet.
- **`LoadPreset` then `SetPresetLevel` in the SAME connection → the level set is overridden** by the load's own level-apply. Set `presetLevel` on a fresh no-load connection; the loaded preset stays "current" across USB reconnects.
- **`LoadPreset` then `LoadScene` in the SAME connection → the load likewise overrides the scene** (symmetric with the level-set gotcha above): send `loadScene` on a separate connection after the preset load — the companion keeps the load+scene combo on its classic two-connection path for this reason.
- **`SetPresetLevel.presetLevel` is a linear amplitude control**: captured loudness ≈ `20·log10(presetLevel) + C` (verified ≤0.04 LU), so preset-to-preset leveling to a target LUFS is one-shot, not iterative. **The active amp's `outputLevel` is ALSO linear/one-shot (HW 2026-06-09, fw 1.8.45): ~25 LU authority, `20·log10(outputLevel) + C` — so per-scene leveling is a ONE-SHOT solve too (measure once → solve `C` → set), NOT the closed-loop trust-region "BatchedLive" path, which was a workaround for a measurement bug (next bullet). `presetLevel` is a GLOBAL multiplier over all scenes → level Base first; per-scene amp values isolate via Scene Edit (`SetNodeSceneEdit`).** The old `~−21 LUFS` target clamp is not universal: maxed 65 Twin / 65 Deluxe NBC reached `−14 LUFS` and clamped around `−8.6 LUFS`. See `apps/tmp-companion/CLAUDE.md`.
- **Re-amp loudness MEASUREMENT must be ISOLATED — one fresh re-amp capture (fresh stream pair) per measured point.** A SHARED continuous re-amp stream read with sliding windows (the BatchedLive approach) mis-measures scenes: it read `−6.96 LUFS` for a scene whose true loudness at that knob value was `~−23/−33 LUFS` (impossible), feeding a closed loop garbage and making it clamp. Isolated capture (`measure_knob_at` shape) reads correctly — which is why per-scene leveling works one-shot. The capture WINDOW is also load-bearing: the full 6 s stimulus + 0.8 s tail. Presets aren't stationary (delay/reverb buildup + decay tail the gated integrated includes), so any early-exit/tail-drop/preroll-skip is a ≤0.3 LU re-baseline, not a free speedup (HW-A/B'd; clean +0.30 / delay +0.16 / reverb −0.02 LU). Validate against the full-capture oracle, not a same-method round-trip (self-consistent → hides the offset). See `apps/tmp-companion/CLAUDE.md`.
- **Block / level-control discovery on fw 1.8.45 must use a RICH lean session, never `connect_for_discovery` (field-78).** field-78 yields no usable field-3 on 1.8.45, so `current_preset_blocks()` over it returns ZERO candidates — silently breaking any leveling built on it ("scenes not changing on the unit"). Use heartbeat warmup → `send_and_collect(LoadPreset)` → pump → `current_preset_blocks()` from accumulated push bodies (companion `discover_blocks_rich` / `prepass_scene_docs`).
- **`LoadPreset` uses field 6 = `presetSlot`**, not field 1. Field 1 is `tabEnum` (0=NotSet, 1=UserPresets, 2=FavoritePresets, 3=CloudPresets, 4=FactoryPresets, 5=Songs, 6=Setlists). The **1-based** `presetSlot = list index + 1` rule (next bullet) also holds for the **Factory bank** (`tabEnum=4`) — HW-confirmed loading factory presets by list-index+1 (Cloud/Favorites banks unverified). The `presetListRequest{listEnum=4}` factory list is enumerable and the handshake already requests it.
- **`PresetListResponse.record[]`** is a `repeated PresetListRecord { string displayName }` — slot is the array index (0-based); no preset ID is sent on the wire. The empty-slot marker is `--` (also "Empty"). ⚠️ **Device userSlot = this list index + 1** (RESOLVED 2026-05-27, fw 1.7.75): `loadPreset.presetSlot`, `saveCurrentPreset.userSlot`, `clearUserPreset.userSlot`, `moveUserPreset.old/new`, AND `songPresetListResponse.userPresetSlot` are all **1-based** while the list is 0-based — add 1 when passing a list index to any slot-addressed setter. (Fixed a latent off-by-one in the auto-leveller, which had passed the 0-based index straight through.)
- **Preset CRUD setters** (byte-exact from a Pro Control capture, firmware 1.7.75; all single-packet, no `batchStatus`): `renameCurrentPreset` (PresetMessage **13**) `{displayName=f1}`; `moveUserPreset` (**16**) `{oldSlot=f1, newSlot=f2}` (persists alone); `clearUserPreset` (**15**) `{userSlot=f1}` = **device** slot. **A Pro Control "rename" is `rename(13)` + `saveCurrentPreset(14){slot=f1}` ~10 ms later** — rename(13) alone is not persisted; send both.
- **`importPresetRequest`** (PresetMessage **117**) `{presetJson=f1}`: writes a full preset back. **`presetJson` = LZ4-block(raw `.preset` file bytes)** = `LZ4(XOR_JLD(compact JSON))` — NOT plaintext, NOT XOR-only. No `batchStatus`; sent standalone post-handshake as a chunked `0x33/0x34*/0x35` stream; device gives **no `importPresetResponse(118)` echo** and follows with its own `loadPreset`. An all-literal LZ4 block is a valid trivial encoder (no compressor dep). **It only APPENDS** (files the preset at the next empty slot) — it cannot target/overwrite a slot. In-place edit recipe = import-to-scratch → `loadPreset(scratch)` → `saveCurrentPreset(origSlot)` → `clearUserPreset(scratch)`. ⚠️ But save-over only preserves a Song link if the new content keeps the original `info.preset_id` + the bound scene (see song-binding note below); a wholesale-different preset breaks the link.
- **The engine normalizes the `info` block to a fixed field set on EVERY `saveCurrentPreset`** (HW round-trip, fw 1.8.45): on-device `presetJson` is PLAINTEXT JSON in the SQLite BLOB, and the firmware parses `info` into a typed `PresetInfo` model, re-serializing it to EXACTLY `{author, created_at, displayName, preset_id, product_id, source_id, timestamp, version}`. Extra keys injected into `info` (`comments`/`artist_name`/`byline`/`song_name`, recognized only by the Qt client — or arbitrary unknown keys) SURVIVE import/restore (the blob is stored verbatim) but are DROPPED the moment the preset is edited+saved on the unit or in Pro Control. So presetJson is not a durable channel for host metadata — key a companion sidecar off `info.preset_id` instead.
- **`exportPresetRequest`** (PresetMessage **115**) `{listEnum=f1, presetSlot=f2}`: Pro Control uses it for export-to-file and gets a complete `.preset`, but it returns **nothing to a seize-probe** — PC almost certainly serialises its own in-memory model, not a device round-trip. Treat as "not replayable as a full USB read"; for a complete preset use the OFFLINE `.preset` codec. **`requestNextEmptyPresetSlot` (PresetMessage 81 → `nextEmptyPresetSlotResponse` 82) is likewise DEAD on 1.7.75** (no reply, standalone or in-burst) — find empty slots by observation (import then re-list).
- **Song-preset binding** (`assignSongPreset` / `songPresetListResponse` records, SongMessage field **11**): a Song row binds a preset by `userPresetSlot` (1-based device slot) + `presetSceneSlot` (scene index) — NOT by `info.preset_id`. The Song read (`songPresetListRequest` 12 → response 13) needs a `batchStatus` and must ride the handshake burst (like preset reads). Because the binding is positional + identity-sensitive, overwriting a slot with a different-identity preset (changed `preset_id` / missing the bound scene) **breaks the Song link** — slot-preservation ≠ link-preservation.
- **Song + setlist CRUD over USB** (captured byte-exact from the device + HW-validated 2026-06-02). Songs = `SongMessage` (TMS field **11**): `addSong[4]{songName=1}`, `renameSong[7]{songSlot=1,songName=2}`, `removeSong[6]{songSlot=1}`, `songNotes[22]{songSlot=1,songNotes=2}`, `setSongBpmActive[23]{songSlot=1,active=2}`; list read `songListRequest[2]`→`songListResponse[3]`, record `{songName=1,songNotes=2,songBpmActive=3,songBpm=4}` (songName capped 24 chars, accents folded to ASCII). Setlists = `SetlistMessage` (TMS field **12**): `addSetlist[4]{setlistName=1}`, `renameSetlist[7]{setlistSlot=1,setlistName=2}`, `removeSetlist[6]{setlistSlot=1}`, `addSetlistSong[14]{setlistSlot=1,songSlot=2}`; list read `setlistListRequest[2]`→`setlistListResponse[3]` (record `{setlistName=1}`), membership `setlistSongListRequest[12]{setlistSlot=1}`→`setlistSongListResponse[13]` (records `{songSlot=1}`). Setlist-membership edit (HW-confirmed 2026-06-02): `removeSetlistSong[16]{setlistSlot=1,setlistSongSlot=2}` and `moveSetlistSong[15]{setlistSlot=1,oldSetlistSongSlot=2,newSetlistSongSlot=3}` address the song by its **1-based POSITION within the setlist** (`setlistSongSlot`, NOT the global `songSlot` that `addSetlistSong` takes); `moveSetlistSong` is array-splice (remove at old, insert at new): `[1,3,6]` remove pos 2 → `[1,6]`; `[1,6,11]` move 3→1 → `[11,1,6]`. All writes are fire-and-forget setters (NO `batchStatus`); list reads carry `batchStatus` + ride the handshake burst. Write slots are POSITIONAL (read-list 1-based index); a new song lands at protocol list slot 1 (shifting every other song +1) but shows 2nd-to-last on the device screen — resolve a just-created song BY NAME.
- **Per-song BPM has NO dedicated setter** — it's the GLOBAL tap-tempo applied to the ACTIVE song: `SettingsMessage[3].tapTempoBpm[12]{value(1)=float, originatorId(2)=1}` (byte-exact `1a09 6207 0d<f32le> 1001`). To set song X's BPM: (1) X needs ≥1 footswitch (`assignSongPreset`), (2) activate X via `presetMessage[2].loadPreset[10]{tabEnum=5, songSlot=4, songPresetSlot=5, presetSlot=6}` (`tabEnum` 5=Songs, 6=Setlists; `LoadPreset` also has `setlistSlot=2`, `setlistSongSlot=3`), (3) send `tapTempoBpm`. The first load after a fresh assign often doesn't settle — retry load+tempo, verifying the target song's BPM by name in a re-read.
- **`CurrentPresetDataChanged.presetJson`** (bytes field 1) is **LZ4-block compressed** JSON (not the LZ4 frame format — no `04 22 4d 18` magic, length comes from the protobuf wrapper). The "stray `0xE8` prefix" earlier observers noted is just the first LZ4 token (`LL=14, ML=8` — 14 literal bytes happen to be the ASCII run `{"ampControl":` that starts every preset). `AllBlockPresetsResponse.field1` (PresetMessage.field136) is also LZ4-compressed. `ProductProfileResponse.field1` (PresetMessage.field42), in contrast, is plain pretty-printed UTF-8 JSON starting `{\n\t"effectCategories":`. Decoder: a pure-Python LZ4-block decode (~30 lines). Try LZ4 first, fall back to a literal block.
- **The decoded `presetJson`** contains `audioGraph.guitarNodes.G1[]` — an ordered list of effect blocks with `FenderId`, `dspUnitParameters`, `nodeId`, `nodeType`. This is the ordered signal chain the companion renders and edits (the Copy feature's `copyModel.ts` → `copy_apply` → `audiograph.rs` path).
- **Unsolicited pushes require a DENSE (~250 ms) `ConnectionHeartbeat` — the cadence IS the subscription (RESOLVED 2026-06-04; supersedes the earlier "no unsolicited pushes" reading).** At a slow heartbeat (~10 s) the session lapses and unit-side preset/scene changes produce nothing past the heartbeat ACKs; at Pro Control's ~4/sec cadence the device pushes `PresetLoaded`(11), `SceneLoaded`(102), `CurrentPresetInfoChanged`(22), `sceneListResponse`(125), and the field-3 preset JSON on every change — there is NO separate subscription message. A live dense session can also EXECUTE fire-and-forget commands inline (`LoadPreset`/`LoadScene` via the companion monitor's command lane) without breaking the heartbeat — ~0.2 s vs the ~2 s fresh-connect bookend (software-green 2026-06-05, HW-validation pending).

## Test & Loopback messages

TMS-native diagnostic / factory surface, all under `TestMessage` (the top-level oneof category, `TestMessage.proto`). 51 sub-types in total; the four most general-purpose are below. The rest cover XMOS programming, SD card test, rear-panel pad / phantom / instrument-select / impedance switching, CPU temperature, brightness, MIDI output — see `TestMessage.proto:55-108` for the full catalog.

| Message                 | Request fields    | Response                                         | Use case                                                         |
| ----------------------- | ----------------- | ------------------------------------------------ | ---------------------------------------------------------------- |
| `LoopbackTest`          | `string data = 1` | server echoes the same `LoopbackTest` back       | generic HID round-trip parity / latency check                    |
| `MidiLoopbackRequest`   | `bool dummy = 1`  | `MidiLoopbackResponse { uint32 statusEnum = 1 }` | MIDI loopback hardware test (rear-panel MIDI in→out)             |
| `SystemInfoRequest`     | (no fields)       | `SystemInfoResponse`                             | firmware/hardware version reporting                              |
| `CpuTemperatureRequest` | (no fields)       | `CpuTemperatureResponse`                         | thermal telemetry — pair with high-load presets for parity tests |

All are `TestMessage`-routed (top-level oneof field 1); the request carries no `batchStatus` echo requirement. None have a Mustang-LT-style `QASlots*` equivalent — TMS rolls factory tests under `TestMessage.testControl` / `testOutput` / `setTestAppMode` (`TestMessage.proto:59,58,100`) instead of an LT-style QA slot model.

See `references/protocol-lineage.md` for how the test surface compares to LT's `LoopbackTest` + `QASlots*` family.
